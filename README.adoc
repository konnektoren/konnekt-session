= Konnekt Session
:toc: left
:toclevels: 2

A Rust library for serverless P2P multiplayer lobby sessions.
Peers connect via WebRTC (Matchbox signalling), share a host-signed event log, and derive consistent lobby state without a central game server.

== Crates

[cols="1,3", options="header"]
|===
|Crate |Purpose

|`konnekt-session-core`
|Domain model — `Lobby`, `Participant`, `Activity`, `DomainCommand`, event sourcing

|`konnekt-session-p2p`
|P2P transport — Matchbox WebRTC, signed event broadcast, `SessionLoop`

|`konnekt-session-yew`
|Yew UI components — `SessionProvider`, `ParticipantList`, `ActivityList`, `LobbyView`, …

|`konnekt-session-cli`
|CLI host/guest tool for terminal-based sessions

|`konnekt-session-tests`
|Integration tests across crates
|===

== Quick Start

=== Yew Frontend

[source,bash]
----
# Run the full lobby UI
SIGNALLING_SERVER=wss://match-0-7.helsing.studio \
  trunk serve --features basic-lobby examples/basic-lobby.rs

# Run the component preview gallery
trunk serve --features preview preview.html
----

=== CLI

[source,bash]
----
# Create a host session
cargo run -p konnekt-session-cli -- create-host --name Alice

# Join an existing session
cargo run -p konnekt-session-cli -- join --session <SESSION_ID> --name Bob
----

== Architecture

=== P2P Flow

Peers connect through a Matchbox signalling server for WebRTC handshake only.
All game messages travel peer-to-peer after that.

[source]
----
Browser A (Host)          Matchbox (Signal)       Browser B (Guest)
     |                         |                         |
     |── offer / ICE ─────────►|── forward ─────────────►|
     |◄─────────────────────── answer / ICE ─────────────|
     |                         |                         |
     |◄══════════ WebRTC DataChannel (P2P) ══════════════►|
     |                                                    |
     | LobbyEvent { host_sig, payload }  ────────────────►|
     |◄──────────────────────────── LobbyEvent  ──────────|
----

=== Key Design Decisions

[cols="1,3", options="header"]
|===
|Decision |Rationale

|No game server
|State lives in the event log; the signalling server only brokers WebRTC handshakes (ADR-0003)

|Ed25519 identity
|Keys derived client-side from name + password — same credentials reconnect to the same identity (ADR-0004)

|Event sourcing
|Lobby state is an append-only event log signed by the host; peers replay to reach consistency (ADR-0010)

|Dual event loops
|P2P transport loop and domain logic loop run independently, communicating via channels (ADR-0020)

|Yew + WASM
|Compile-once Rust runs natively in the browser with full type safety (ADR-0001, ADR-0002)
|===

See `konnekt-session-workspace/adr/` for all 23 Architecture Decision Records.

== Yew Components

Components are mounted under a `SessionProvider` which owns the P2P session loop.

[source,rust]
----
html! {
    <SessionProvider signalling_server="wss://match.example.com" name="Alice">
        <LobbyView />
    </SessionProvider>
}
----

=== Component Preview

The `preview` feature adds a `yew-preview` gallery for all UI components. This interactive browser allows testing various component states and variants without a backend.

[source,bash]
----
# Run from the konnekt-session-yew directory
trunk serve --features preview preview.html
----

Groups:

* *Session* — `SessionInfo` (host / guest / solo variants)
* *Lobby* — `ParticipantList`, `ActivityList`
* *Activity* — `ResultsView`, `SubmissionStatus`, `ActivitySubmission`
