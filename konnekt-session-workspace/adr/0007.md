# ADR-0007: Use Cucumber for Behavior-Driven Testing

**Status**: Accepted | **Date**: 2025-12-29

## Context

Gap between DDD docs (AsciiDoc) and tests (Rust). Domain events from Event Storming had no executable representation. Non-technical stakeholders couldn't verify behavior.

## Decision

Use **`cucumber-rs`** with **Gherkin** syntax for executable specifications.

## Why

- Direct mapping: Event Storming event → Gherkin `Scenario` → step definition
- Ubiquitous language enforced: "Host", "Guest", "Active Mode" in test files
- Living documentation — tests fail if behavior diverges from specs
- Natural multi-peer scenario expression

## Feature Organization

```
features/
├── lobby/     ← join_lobby, host_delegation
├── participant/ ← active_mode, spectator_mode
├── activity/  ← start_activity, submit_result
└── p2p/       ← connection, reconnection
```

## Traceability Chain

```
Business Need → Domain Event → Feature → Test → Code
```

## Trade-offs

- ✅ Stakeholder-readable, living docs, P2P scenarios natural
- ❌ Additional test layer on top of unit tests
- ❌ Slower than unit tests, learning curve for Gherkin
