# ADR-0008: Use thiserror for Domain Errors

**Status**: Accepted | **Date**: 2025-12-29

## Context

Need type-safe errors across domain (business rules), infrastructure (P2P, signatures), and presentation layers. Errors must be pattern-matchable and serializable for P2P transmission.

## Decision

Use **`thiserror`** for all error types.

## Why

- Derive macro generates `Display` + `Error` traits — zero boilerplate
- Pattern-matchable: `Err(LobbyError::Full { capacity })` → specific UI message
- `#[from]` enables automatic error chaining across layers
- Domain errors derive `Serialize`/`Deserialize` — sent to peers as typed rejection reasons
- Clear layer boundaries: `LobbyError`, `P2PError`, `SerializationError`

## Error Organization

| Layer | Types |
|-------|-------|
| Domain | `LobbyError`, `ParticipantError`, `ActivityError` |
| Infrastructure | `P2PError`, `SerializationError`, `SignatureError` |
| Application | wraps domain/infra, converts to user messages |

## Rules

- ✅ Derive `Clone` + `PartialEq` on domain errors (testability)
- ✅ Derive `Serialize`/`Deserialize` on P2P errors
- ❌ Never use `panic!()` or `unwrap()` in production code
- ❌ Never use plain `String` errors in domain layer

## Trade-offs

- ✅ Compile-time exhaustive matching, zero boilerplate
- ❌ ~20KB WASM binary size, compile time from proc macros
