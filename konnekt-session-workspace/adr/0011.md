# ADR-0011: Use std::future for WASM-Compatible Async

**Status**: Accepted | **Date**: 2025-12-29

## Context

Async needed for P2P, UI, storage, timers. Code must run in both WASM (browser) and native (tests). Tokio doesn't work in WASM (requires OS threads, adds ~200KB).

## Decision

Use **`std::future::Future`** as the async primitive. Execute with **`wasm-bindgen-futures`** in WASM and **`tokio`** in tests.

## Execution Split

```rust
#[cfg(target_arch = "wasm32")]
pub fn spawn<F: Future<Output=()> + 'static>(f: F) {
    wasm_bindgen_futures::spawn_local(f);
}

#[cfg(not(target_arch = "wasm32"))]
pub fn spawn<F: Future<Output=()> + Send + 'static>(f: F) {
    tokio::spawn(f);
}
```

## Timers

- WASM: `gloo-timers` (`sleep(Duration::from_secs(30)).await`)
- Tests: `tokio::time`

## Why

- Zero runtime in WASM production bundle — uses browser's native event loop
- Business logic stays runtime-agnostic (no `tokio::spawn` in domain)
- Full Tokio ecosystem available in tests

## Trade-offs

- ✅ Minimal WASM size (~5KB for `wasm-bindgen-futures`), standard Rust async
- ❌ Two execution paths, some `#[cfg]` blocks needed
