# ADR-0017: Use Schemars for Schema Generation with Aide for OpenAPI

**Status**: Accepted | **Date**: 2025-12-29

## Context

Need schema documentation for P2P protocol, TypeScript bindings, potential HTTP APIs. Single source of truth: code defines the schema.

## Decision

Use **`schemars`** for JSON Schema generation on all domain types. Use **`aide`** as optional feature for OpenAPI when HTTP APIs are needed.

```rust
#[derive(Debug, Clone, Serialize, Deserialize, JsonSchema)]
pub struct Participant {
    #[schemars(length(min = 1, max = 50))]
    #[schemars(example = "Alice")]
    pub name: String,
    pub lobby_role: LobbyRole,
    pub participation_mode: ParticipationMode,
}
```

## Why Schemars

- Reuses `serde` annotations — no duplicate type info
- JSON Schema is the interchange standard (→ TypeScript, OpenAPI, GraphQL)
- ~60KB WASM overhead
- Validation in tests via `jsonschema` crate

## Why Not utoipa

utoipa duplicates serde: requires `#[schema(value_type = String)]` alongside `#[serde(rename)]`. Schemars reads serde annotations directly.

## Workflow

```
schemars → JSON Schema → {TypeScript, OpenAPI, docs}
```

## Trade-offs

- ✅ Single source of truth, standard format, tool ecosystem
- ❌ Indirect OpenAPI (need aide or manual conversion)
- ❌ Two tools: schemars + aide (aide is optional)
