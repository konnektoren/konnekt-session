# ADR-0005: Use Tracing for Logging and Diagnostics

**Status**: Accepted | **Date**: 2025-12-29

## Context

Need structured logging with async context propagation. Runs in WASM (browser) and native (tests). Browser console, no file system.

## Decision

Use **`tracing`** with **`tracing-wasm`** for browser and **`tracing-subscriber`** for native.

## Why

- Structured key-value fields (filterable in DevTools: `peer_id`, `lobby_id`)
- Spans track operation lifecycle across async boundaries (`.instrument(span)`)
- Zero-cost: disabled log levels compiled away entirely
- Browser DevTools shows spans as collapsible console groups
- Future-proof: swap in `tracing-opentelemetry`, metrics, Sentry

## Log Level Convention

| Level | Use |
|-------|-----|
| `ERROR` | Unrecoverable, user-facing |
| `WARN` | Recoverable, degraded |
| `INFO` | Significant state changes |
| `DEBUG` | Diagnostic detail |
| `TRACE` | Very verbose internals |

## Trade-offs

- ✅ Rich context, span hierarchy, async transparency
- ❌ ~30KB WASM overhead vs ~5KB for `log` crate
- ❌ Steeper learning curve than `log::info!()`
